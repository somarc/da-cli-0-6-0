f023 · migrate, contract · fixed

migrate import accepted a path it could never serve — and said "imported"

Symptom

Wave 4 migration rep: migrate import --path /migrated/example uploaded cleanly and printed status: 'imported'. But the explicit --path was used verbatim — no .html extension — so the resulting DA source could never be listed, previewed, or published. Derived paths (from the source URL) got ensureHtmlPath normalization; explicit paths skipped it entirely. The strictest, most intentional input got the weakest handling.

Why it matters

This is f021's lesson wearing a different coat: false success is the worst contract violation. An agent migrating a hundred pages with explicit paths would see a hundred green envelopes and a site with zero servable routes. The command that exists to move content into DA produced content DA cannot serve — silently.

Resolution

Fixed in da-cli: explicit --path values pass through the same ensureHtmlPath normalization as derived ones. Field proof: the same import re-run lands at /migrated/example.html, is listable, previews clean, and migrate validate matches the known source (1 heading, title "Example Domain", 1 external link). Evidence: dogfood/evidence/wave-4/migrate-import.json, migrate-validate-import.json.

Lock

migrate-paths.test.js asserts explicit-path normalization alongside the existing derived-path suite: extension-less input gains .html, already-correct input passes through untouched, and non-HTML asset paths are left alone.
