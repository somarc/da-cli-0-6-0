f016 · code, interop · fixed

code job had never worked — wrong route shape, imaginary terminal states

Symptom

Found by the release-review subagent audit (interop lane), verified against production: da code job <id> called /job/{id}/details, but the real Helix Admin route is /job/{org}/{repo}/{topic}/{jobName}/details — an unconditional 404 on every invocation since the command shipped. Compounding it, helixJobWait polled for terminal states failed/success, which do not exist in the platform's state machine (created / running / stopped / cancelled / expired) — a cancelled job would spin the full 120s timeout.

Why it matters

A command that 404s on 100% of invocations survived because nothing exercised it — the same coverage-gap predictor as f015. And polling for states the platform never emits turns every wait into a guaranteed timeout: the worst kind of interop bug, where the code agrees with the docs in its head instead of the wire.

Resolution

Fixed in da-cli: code job calls the real route (with --topic support), and helixJobWait treats stopped/cancelled/expired as terminal — ground truth from helix-api-service source and live admin.hlx.page responses, not documentation folklore.

Lock

URL-shape and state-machine tests added; two pre-existing locks that had encoded the broken URL and the imaginary states were flipped to assert ground truth. A test that locks the wrong behavior is worse than no test — it certifies the bug.
