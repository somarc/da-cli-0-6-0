f015 · job, safety · fixed · CRITICAL

job run executed live publishes with no --commit check at all

Symptom

Found by the release-review subagent audit (safety lane), reproduced live: the --commit gate existed only at job creation. Execution (job run, job start, detached workers) trusted unsigned local JSON in .da/jobs/ as carried-forward intent — a crafted or stale job file executed real preview/publish mutations with no gate anywhere. Also reachable via pipeline YAML: job was in ALLOWED_PIPELINE_COMMANDS but missing from commandIsMutating().

Why it matters

This falsified the CLI's central safety promise — every mutation behind root --commit — for an entire command family. The predictor was visible in coverage: job.js had zero tests, and the audit chose its targets by exactly that signal. Untested command files are where the critical bugs live.

Resolution

Fixed in da-cli: the gate now runs at execution time — job run/job start without root --commit leave the job pending and print the dry-run message. Detached workers forward commit intent explicitly. job, index, and sitemap added to the pipeline mutation pre-guard so YAML cannot route around it.

Lock

First-ever job.test.js reproduces the exact attack: a crafted job file run without --commit must stay pending; with --commit it executes. Pipeline pre-guard locked alongside.
