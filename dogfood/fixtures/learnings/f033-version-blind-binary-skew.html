f033 · provenance, status · fixed

the binary that answered was not the tree we fixed

Symptom

A field session shipped a content update through PATH da and observed publish page --format json succeed with empty stdout — filed on sight as a fresh f017 specimen. It wasn't. The binary on PATH was a fifteen-day-old global npm install that predated the entire f014–f032 flywheel; the current tree had long since fixed the bug. Both installs reported the same version — 0.5.1 — because the package deliberately holds its version until the 0.6.0 gate opens. Nothing on the machine channel could distinguish the stale install from the fixed one.

Why it matters

A dogfood run whose evidence is generated by a stale binary is measuring the past. Ghost findings burn flywheel cycles on bugs that are already dead — and the same skew works in reverse: a stale binary silently hides every new fix and regression the tree carries. The version string is the natural place to look and it is exactly the signal that cannot work on a hold-the-version line. Provenance has to be observable, not inferred: which install answered? is a question the tool itself must answer.

Resolution

Fixed in da-cli: da status now reports data.cli — the running install's packageRoot, installKind (installed-package vs source-tree), version, and for source trees the exact git sha + dirty flag. One status call at session start makes binary skew visible: a two-week-old artifact and a current worktree no longer read identically. Machine remediation: the stale global install was replaced with an npm link to the working tree, so PATH da is the tree the flywheel fixes. Closed in the same pass: the f021-residue publish pages variadic asymmetry (real in the current tree — the promote step refused the exact page list the preview step accepted) plus its missing dry-run envelope.

Lock

Provenance lock in status.test.js (packageRoot named, source-tree classification, sha shape, dirty boolean, envelope carries data.cli) and variadic-parity lock in publish.test.js (two direct paths dry-run to one publish.pages envelope listing both normalized paths). Both verified failing on old code; 611/611. Field proof: da status from any directory now names the linked tree at its exact commit; the same two-path publish that refused with "too many arguments" promotes 2/2. Residue: committed preview pages/publish pages success output is still a bare array under json — on the f017 march list.
