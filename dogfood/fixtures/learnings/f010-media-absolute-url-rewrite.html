Relative /media/… image URLs rewrite to about:error

Symptom

DA HTML authored with host-less image paths:

<img src="/media/example.webp" alt="">

After preview, .plain.html showed <img src="about:error"> even when the binary existed on the content bus and GET /media/example.webp redirected to a hashed media URL successfully.

Absolute content-bus URLs worked:

https://content.da.live/{org}/{repo}/media/example.webp

Why it matters

Agents default to site-relative media paths. Failures are silent in the browser (broken image) and only obvious in plain HTML. Video <a href="/media/….mp4"> links often still work because they are not Media-Bus-rewritten the same way — asymmetric and confusing.

Workaround (today)

  1. Put the binary with da content put /media/….
  2. Reference images with absolute https://content.da.live/{org}/{repo}/media/… (or a reachable absolute URL) so preview can fetch and hash into Media Bus.
  3. Re-put and re-preview the HTML after media is on the bus.

Resolution (planned)

Product options: (a) Helix/DA preview resolves same-tenant host-less /media/* against the content bus before Media Bus rewrite; (b) da content put / a content linter warns when HTML contains host-less img src; (c) agent skill default remains absolute URLs until (a) ships. Prefer (a)+(b) for low-friction agents.

Lock

Pending platform/CLI. Observed on multi-site dogfood (hero plates, poster stills). Skills should keep absolute-URL guidance until fixed.
