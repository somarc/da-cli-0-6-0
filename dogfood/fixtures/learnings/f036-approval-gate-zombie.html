f036 · pipeline, contract · fixed

the approval gate exited 0 over an unanswered question

Symptom

Drill D3's approval rep: a pipeline step with requires_approval: true, run with stdin closed — the standard shape for every orchestrated, piped, or CI execution. The readline prompt's callback can never fire on EOF, the event loop drained, and the process exited 0 mid-run with no envelope — while the run state file stayed running forever with zero step results. An orchestrator sees success; pipeline status sees a run still going. Both are wrong. And on the answered path, the prompt printed to stdout, prefixing the json envelope with prose — f014's channel lesson at a new gate.

Why it matters

An approval gate is a safety primitive: it exists so a mutation cannot proceed without a decision. This one inverted its purpose under exactly the conditions agents run in — the undecided gate produced a success exit and a zombie state, the two lies compounding: nothing failed, and nothing finished. f020 taught this shape once (implicit interactive auth hanging non-TTY runs); the gate had not learned it.

Resolution

Fixed in da-cli: approvals are now explicit where interaction is impossible. pipeline run --approve <step-id…> / --approve-all grant gates in any context; a non-TTY run that reaches an ungranted gate refuses immediately — one ok:false envelope naming the step and the exact flag to pass, run state terminal failed, exit 1. The TTY prompt writes to stderr (stdout stays a single envelope), and EOF during a prompt is denial, not limbo. The denial path was already contained (step held at pending, f025's abort semantics intact); it now has a clean channel too.

Lock

Two CLI locks in pipeline.test.js under the hermetic riverboat harness: closed-stdin run of a gated step → exit 1, ok:false envelope matching --approve gated, run status failed (no zombie); --approve-all → exit 0, step completed, stdout parses as exactly one envelope (a leaked prompt would break the parse). Both verified failing on old code; 618/618. Field proof: the same drill YAML that stranded run 9f98d66c as a zombie now refuses in milliseconds with the recovery flag named, and completes cleanly under --approve-all.
