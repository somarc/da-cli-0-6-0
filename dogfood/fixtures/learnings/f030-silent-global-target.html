f030 · resolve, safety · fixed

the wrong-directory agent read the wrong site — and the CLI said nothing

Symptom

Wave 5 drill A1 (unresolved target), first probe: from an empty directory with no project config at all, content list / didn't refuse — it silently resolved to the GLOBAL default target and listed a different project's content with warnings: []. An agent in the wrong cwd reads, reasons about, and plans against the wrong site with zero signal. Second probe, truly unresolved (no config anywhere): the refusal envelope was correct — but next: [] was empty, with the recovery buried in error prose.

Why it matters

f019 protects mutations against config-derived targets; reads had no equivalent orientation signal, and reads are what agents act on. Silent cross-project fallback is the read-side twin of f001's original drift. And next: [] on a refusal breaks the contract's promise: executable recovery lives in next[], not inside prose an agent must parse.

Resolution

Fixed in da-cli: when org AND repo both resolve from global config (no flags, no project .da.json), the client warns on stderr and the read-listing envelopes carry the warning: "Target …/… resolved from GLOBAL config — pass explicit flags or run da site pin-target". Unresolved-target errors now attach executable err.next (da --org <org> --repo <repo> <command>, da config set org <org>), honored by the envelope seam. Explicit flags stay silent — no noise for correct usage.

Lock

Three CLI-level locks in content-envelope.test.js: global-fallback carries the orientation warning; explicit flags carry none; truly-unresolved refuses with populated next[]. Verified failing on old code; 609/609.
