f014 · pipeline · fixed

pipeline run --format json let child-step stdout pollute the envelope

Symptom

Found during the Wave 4 opening rep: a full committed construct.yaml regeneration (58 steps) completed perfectly — every step completed, exit 0, all 28 public routes byte-identical before and after. But the parent's --format json stdout was unparseable: 29 lines of child-step output (preview URLs) printed above the JSON envelope. The runner spawned each step with stdio: 'inherit', so child stdout interleaved with the parent's structured output stream.

Why it matters

This is a direct violation of the first trust commitment — trust the contract. An agent running the flagship coordination surface does JSON.parse(stdout) and fails on a flawless run. The envelope was present and correct at the end of the stream; the contract broke anyway, because stdout was shared. A structured-output mode owns its stdout or it is not a structured-output mode.

Resolution

Fixed in da-cli: when the parent pipeline runs with --format json, child-step stdout is piped to the parent's stderr instead of inherited — progress stays visible to a human watching the run, and stdout carries exactly one envelope. Non-JSON (table/md) behavior is unchanged. One seam covers both the da-subcommand and riverboat execution paths (spawnCommand in pipeline-runner.js).

Lock

End-to-end tests in pipeline.test.js: a real pipeline with a stdout-polluting step run under --format json must produce parseable stdout with the pollution on stderr; the same pipeline without JSON format must still inherit stdout. 524/524 suite passing. Field proof: post-fix construct re-run parses clean; evidence under dogfood/evidence/wave-4/.
