f012 · auth, content · fixed

DA permission-hint headers discarded; 403 has no distinct handling

Symptom

DA's admin API returns x-da-actions / x-da-child-actions response headers hinting at the resolved permission set for a path (and for a folder listing's children). src/lib/da-client.js discarded response headers on every fetch path, including on 200 success — only status, url, and the raw text body were ever captured (see DaApiError). Separately, src/lib/api-error.js's handleApiError special-cased 401 ("run da auth login") and 404 ("Not found"), but 403 fell into the generic else branch and printed the raw DA API 403 at <url>: <body> message — identical treatment to a network blip or any other unexpected status.

Why it matters

An agent calling da content list or da content get had no way to learn "you have read-only access here" until it attempted a write and failed. And when a write did fail with 403, the agent got no signal distinguishing "you're authenticated but not permitted" (recoverable by requesting access) from "you're not authenticated" (recoverable by da auth login) or from a transient error. This directly undermined the ADR 0001 orientation principle — "every operation should leave the agent better oriented than before it ran."

Resolution

Fixed in da-cli (branch fix/f012-da-permission-headers, not yet merged):

  • DaApiError now captures resolved permission hints from response headers via extractPermissionHints(), exposed as err.permissionHints
  • client.list() attaches the same hints to its returned data as a non-enumerable daPermissionHints property (existing consumers unaffected)
  • handleApiError gives 403 a distinct message — "Permission denied," notes da auth login will not fix it, and includes the resolved access hint when present
  • content list / content get proactively warn() to stderr when resolved access is read-only, before any write is attempted

Honest bound: this is the plain-CLI error path and a proactive stderr warning, not a full agent-v1 envelope permission-denied error code or next[] hint — those commands don't use the envelope contract yet. da resolve was out of scope: it's a pure local URL/path parser and never makes a network call, so there are no permission hints for it to surface.

Lock

Unit tests: da-client.test.js (extractPermissionHints, DaApiError.permissionHints, list() attaching non-enumerable hints), api-error.test.js (403 message distinct from 401/generic, includes resolved-access hint), content.test.js (warnIfReadOnly: silent when hints absent or include write, warns with resolved actions when read-only). 522/522 suite passing.
