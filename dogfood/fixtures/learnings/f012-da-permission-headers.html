f012 · auth, content · open

DA permission-hint headers discarded; 403 has no distinct handling

Symptom

DA's admin API returns x-da-actions / x-da-child-actions response headers hinting at the resolved permission set for a path (and for a folder listing's children). src/lib/da-client.js discards response headers on every fetch path, including on 200 success — only status, url, and the raw text body are ever captured (see DaApiError). Separately, src/lib/api-error.js's handleApiError special-cases 401 ("run da auth login") and 404 ("Not found"), but 403 falls into the generic else branch and prints the raw DA API 403 at <url>: <body> message — identical treatment to a network blip or any other unexpected status.

Why it matters

An agent calling da resolve or da content list today has no way to learn "you have read-only access here" until it attempts a write and fails. And when a write does fail with 403, the agent gets no signal distinguishing "you're authenticated but not permitted" (recoverable by requesting access) from "you're not authenticated" (recoverable by da auth login) or from a transient error. This directly undermines the ADR 0001 orientation principle — "every operation should leave the agent better oriented than before it ran" — and the agent-v1 envelope's structured error-code taxonomy, which has no code for permission-denied today.

Resolution

Not yet fixed. Candidate approach: capture x-da-actions / x-da-child-actions on every response in da-client.js, surface resolved actions in resolve / content list / content get envelope data, and give handleApiError (plus the agent-v1 envelope error path) a distinct permission-denied code for 403 with a next[] hint (distinct from 401's auth login hint).

Lock

Not yet written — pending fix. Will need a unit test asserting 403 produces a distinct error code/message from 401/generic errors, and a resolve/list fixture asserting permission hints surface in envelope data when present.
