f025 · pipeline, durability · fixed

pipeline abort was a no-op in practice — the runner clobbered its own kill switch

Symptom

The Wave 4 interruption drill, pipeline edition: pipeline abort writes _meta.status: 'aborted' into the run's state file, and the runner checks that file at every batch boundary. But every per-step save rewrote _meta from memory — so an abort landing while any step was executing (i.e. almost always, that's what "running pipeline" means) was overwritten before the boundary check could see it. The run finished completed as if the abort never happened.

Why it matters

An abort that only works when nothing is running is not an abort. For agents, pipeline abort is the containment primitive for "this 74-step committed regen is doing something wrong, stop it" — and it silently did nothing in exactly that situation. Found by running the drill, not by reading the code: the state file looked correct after abort returned, and was quietly reverted milliseconds later.

Resolution

Fixed in da-cli: every persist now merges the on-disk abort marker into memory first (persistState), so the marker survives step saves and the batch-boundary check honors it. Field proof on the real site regen: abort issued mid-flight → run stops at the boundary with status: 'aborted', 11 steps completed, 63 never ran, exit 1 with one parseable ok: false envelope. One in-flight step (put-blocks) completed 1.8s after the abort and its save preserved the marker — the exact window that used to clobber it. Recovery: a clean rerun converges (idempotence already proven byte-level).

Lock

e2e in pipeline.test.js: a step aborts its own run via the real pipeline abort command; asserts the dependent next-batch step never executes, final status aborted, and abortedAt preserved on disk. Verified failing against the old runner — same pipeline reported completed. Evidence: dogfood/evidence/wave-4/pipeline-abort-envelope.json, pipeline-abort-state.json, pipeline-abort-recovery.json.
