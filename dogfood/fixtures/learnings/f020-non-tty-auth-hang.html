f020 · auth, agent-ux · fixed

No token + no TTY = indefinite hang: CI caught the environment coupling the local suite couldn't

Symptom

The f019 push turned CI red on Node 22 and 24: 8 assertion failures, each hanging exactly 30s with exitCode null !== 1. The new publish.test.js spawned the real CLI in a bare temp HOME, where getToken() found no cache and no DA_TOKEN — and launched npx da-auth-helper with inherited stdin: an interactive, network-fetching auth bootstrap that waits forever in a non-TTY context. The locks had passed locally only because of warm ~/.aem token state.

Why it matters

One bug wearing two hats. Product: any agent, CI job, or pipe invoking a committed command without a token hung indefinitely instead of failing — the worst possible agent UX, worse than any error. Test discipline: the locks were environment-coupled, asserting the runner's auth state instead of the CLI's authority gate. And the hang fired before the f019 safety gate was ever reached — auth bootstrap ordering put an interactive prompt in front of a refusal that needed no auth at all.

Resolution

Fixed in da-cli (two layers): (1) when stdin is not a TTY and no token is cached, getToken() throws immediately with executable recovery — set DA_TOKEN, or run da auth login from an interactive terminal. The 30s hang is now an instant exit 1. (2) The f019/f017 locks are hermetic: synthetic DA_TOKEN, isolated HOME/DA_TOKEN_PATH — they test target authority, never runner auth state. CI back from 4m28s red to 34s green.

Lock

Auth test reproducing the exact bare-HOME, non-TTY condition asserts the fast failure with recovery guidance (skips gracefully in a real terminal). Tracked residue: the fail-fast surfaces as a thrown error, not an ok:false envelope — bootstrap errors flow into the f017 remainder (printErrorEnvelope migration).
