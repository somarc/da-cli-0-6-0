f037 · content, media · fixed

content put committed a page pointing at DA media that did not exist — the broken image was found by a human, not the CLI

Symptom

Shipping the wave certificate pages: a page fixture referenced https://content.da.live/somarc/da-cli-0-6-0/media/renderaissance/parthenon-frame-study.jpg. That asset existed only on the codebus (the git repo's media/ folder) — DA's media store held 14 assets, none of them this file. da --commit content put accepted the page with zero warnings; preview succeeded; the rendered page served <img src="about:error">. The failure surfaced when a human looked at the page.

Why it matters

f010 taught put to guard host-less /media URLs — warn, rewrite, or fail. This is f010's sibling: an absolute content.da.live URL, pointing into the very org/repo the CLI is authenticated against, referencing nothing. The command with the authority and the credentials to check its own project's media chose not to look. The result is the worst failure class in this suite's vocabulary: a wrong outcome committed as success — same family as f023 ("imported" unservable content) and f029 (claimed migration never on the platform). Agents cannot recover from failures that report as clean.

The distinction that makes this checkable: codebus media (/media/… served from git) and DA media (ingested at put, rewritten to ./media_<hash>) are different stores with different existence semantics — and the CLI knows which is which at put time.

Resolution

Field recovery same-session via the binary upload lane: da --commit content put /media/renderaissance/parthenon-frame-study.jpg …, then re-put + re-preview. Verified: src rewritten to ./media_1096636e….jpg, optimized asset serves 200.

Fixed in da-cli: put's media preflight now extends to absolute same-project DA media references — one authenticated HEAD per unique reference (sourceExists, a new client probe where 404 is definitive and transport/auth failures return unknown, so a flaky probe can never block a put). Missing references warn by default on both channels: envelope warnings[] and stderr, naming the paths, the about:error consequence, and the exact upload command. --strict-media-urls extends to fail the put with one ok:false envelope carrying the per-asset upload recovery in next[], exit 1. Cross-project and external URLs are untouched — not this command's authority. Field-proven against the live project: warn envelope on a reference-to-nothing draft, strict refusal with executable recovery, and zero noise on the recovered Wave 4 page (dogfood/evidence/wave-5/f037-fix-field-*.json).

Lock

Hermetic CLI locks via the DA_ADMIN_URL seam (content-put-media.test.js): missing reference → dry-run envelope carries exactly one f037 warning naming path + consequence, stderr carries the upload recovery; strict mode → one ok:false envelope, non-zero exit, next[] executable; existing / cross-project / external references → zero f037 noise. Unit locks for the collector (dedupe, srcset, query-string stripping, authority boundaries). Verified failing on old code (2/3 fail, control passes); suite 625/625.
