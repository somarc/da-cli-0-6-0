f017 · contract, output · partial (seams landed, migration ongoing)

The envelope promise was 9% true — 2 of 23 command groups used the contract

Symptom

Found by the release-review subagent audit (contract lane): only resolve and status emitted the validated agent envelope. content put succeeded silently under --format json; preview page/publish page printed bare URLs; site create interleaved 13 text lines with JSON; config/auth never checked format at all. The field report graded this "trust the contract: B-" — an agent spent a session grepping stderr for ^Uploaded|error because success and failure were both unstructured.

Why it matters

An agent-first CLI is its envelope. If JSON.parse(stdout) works on 2 commands and fails on 21, the contract is a demo, not a contract — and every consumer builds fragile per-command scraping instead.

Resolution

Landed in da-cli: printEnvelope/printErrorEnvelope seams bridging to the validated contract; content put, preview page, publish page emit exactly one envelope under json (dry-run and committed, with next[] steps). The least-friction pass then closed the failure side: handleApiError — the choke point every API failure runs through — emits one ok:false envelope with status-aware recovery commands. Residual f014 gap closed too: emission keys on resolveFormat(), so piped-but-unflagged runs (the default agent pattern) get clean envelopes.

Lock

Unit + e2e locks including the piped-no-flag case; the test that had encoded the resolveFormat gap as expected behavior was flipped. Remaining (tracked): migrate the other ~18 command groups — config/auth/site create are the worst stdout polluters — and route bootstrap errors (e.g. auth failures thrown before command dispatch) through printErrorEnvelope.
