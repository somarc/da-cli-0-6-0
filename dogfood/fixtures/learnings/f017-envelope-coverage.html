f017 · contract, output · partial (seams landed, migration ongoing)

The envelope promise was 9% true — 2 of 23 command groups used the contract

Symptom

Found by the release-review subagent audit (contract lane): only resolve and status emitted the validated agent envelope. content put succeeded silently under --format json; preview page/publish page printed bare URLs; site create interleaved 13 text lines with JSON; config/auth never checked format at all. The field report graded this "trust the contract: B-" — an agent spent a session grepping stderr for ^Uploaded|error because success and failure were both unstructured.

Why it matters

An agent-first CLI is its envelope. If JSON.parse(stdout) works on 2 commands and fails on 21, the contract is a demo, not a contract — and every consumer builds fragile per-command scraping instead.

Resolution

Landed in da-cli: printEnvelope/printErrorEnvelope seams bridging to the validated contract; content put, preview page, publish page, deploy pages, and the platform bulk preview/publish paths emit exactly one envelope under json. The least-friction pass closed API failures through handleApiError. This return to the loop closed the next boundary: errors thrown before command-local catches — reproduced with implicit auth bootstrap in a non-TTY shell — now become one ok:false envelope with the real command operation and executable auth recovery, instead of empty stdout plus a Node stack. Residual f014 behavior also keys on resolveFormat(), so piped-but-unflagged runs get the same contract.

Lock

Unit + e2e locks include the piped-no-flag case and two new child-process locks for pre-dispatch auth failure. The audit also caught a real regression in the supposedly migrated content put dry-run: oldContent was block-scoped, so building its envelope threw; scope is fixed. Remaining (tracked honestly): core non-bulk content/preview/publish/deploy lifecycle paths still include silent, raw, or multi-document JSON, and reporting groups remain on legacy shapes. config/auth/site create remain the worst stdout polluters. The next closure slice is the core delivery write lifecycle, not a false repo-wide victory claim.
