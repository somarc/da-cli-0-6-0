f013 · route · open

route classify probes miss codebus static HTML under /tools/

Symptom

During the Wave 3 cut, da route classify /tools/wave-2-cli-surface.html returned ownership orphan with preview, live, and code probes all reporting 404 — while a plain curl of the same path on the live host returns 200. The page is static HTML checked into git under tools/ and served straight from the code bus; it is code-owned in practice, but the classifier cannot see it.

Why it matters

Orphan is the clean-candidate class. A probe gap that mislabels a living code-bus page as orphan invites route clean toward content that is very much owned — the exact wrong-orientation failure the ownership matrix exists to prevent. Agents must be able to trust that orphan means "no owner," not "the probe didn't know where to look." Extensioned static routes (/tools/*.html, and similar) are a common EDS pattern for sidekick plugins and site tooling, so this is not an exotic corner.

Interim guardrail

Documented on the route matrix at the Wave 3 cut: when class is orphan but the path is known codebus static, prefer curl/live proof and do not treat the route as a clean candidate. A live 200 that disagrees with an all-404 probe is a probe conflict, not an orphan.

Resolution

Open. Candidate fix in da-cli: align the codebus probe for extensioned static routes with how the edge actually serves them (probe the raw path against the code bus in addition to the helix status code.status read), and surface live-200/probe-404 disagreement as an explicit probe-conflict signal instead of falling through to orphan. Severity: Medium — tracked in dogfood/FRICTION-GATE-0.6.0.md; does not block the Critical/High friction column.

Lock

Pending. When fixed, lock with regression coverage in route-classify.test.js (static codebus HTML must not classify orphan) and re-run the Wave 3 ownership matrix so /tools/wave-2-cli-surface.html reports its true class publicly.
