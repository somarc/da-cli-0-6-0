f035 · contract, cli-surface · fixed

the documented off-switch didn't switch anything off

Symptom

Drill B2 (invalid section shape) needed to bypass the strict-sections door deliberately — that was the injection. --no-strict-sections, the documented opt-out, did nothing: the upload still blocked. Commander stores a --strict-sections/--no-strict-sections pair in one camelized key — the negated flag sets strictSections: false, it never creates a noStrictSections key — so the code checked a key that cannot exist and fell through to the DA-backed default: strict. Only the undocumented-in-the-message alias --allow-top-level-blocks worked. Three contract breaks in one refusal: the off-switch was dead, the prose said "omit the flag" when no flag was ever passed, and stdout was empty under json — the exact --strict-sections march residue f032 had already tracked.

Why it matters

A safety gate whose escape hatch is broken teaches operators to distrust gates. An agent that reads --no-strict-sections in the help, passes it, and still gets blocked has no honest move left — the tool's stated contract and its behavior have parted ways, and the refusal's own recovery advice pointed at a flag state that was not the cause. Meanwhile the empty machine channel made the correct refusal indistinguishable from a crash.

Resolution

Fixed in da-cli: resolveSectionStrictness honors the commander shape — strictSections === false (what --no-strict-sections actually produces) disables strictness. The strict refusal now emits one ok:false envelope naming the offending block classes and two executable recoveries: content fix-sections --write (fix the shape) or --allow-top-level-blocks (own the bypass). The prose names the real cause — "section shape is strict by default on DA-backed repos" — instead of blaming an absent flag. Drill context: the flatten this door guards against is invisible to the rendered-side audit (audit full passed 0-errors on the flattened page, because the violation erases its own evidence) — preview explain is the source-aware recognizer that names it (classes-removed-in-preview, exit 1), field-proven in the same drill.

Lock

Lock in content-diagnostics.test.js: resolveSectionStrictness({ strictSections: false }, daBacked=true) === false — the exact field shape, verified failing on old code; 618/618. Field proof: the same fixture that refused under --no-strict-sections now dry-runs clean with it, and the default-strict refusal parses as one envelope with the block names and both recoveries.
