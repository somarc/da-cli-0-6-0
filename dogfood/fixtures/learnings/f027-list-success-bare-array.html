f027 · content, contract · fixed

content list succeeded in a shape agents can't verify — bare array instead of envelope

Symptom

Found in the first minutes of Wave 5, running the expired-auth recovery drill: with the token expired, content list --format json failed correctly — one ok: false envelope, executable recovery (da auth login), exit 1. After recovering, the verification read returned… a bare JSON array. No ok, no operation, no envelope. The agent proving its own recovery could parse the failure but not assert the success. Empty results were worse: nothing on stdout at all. Same asymmetry in content tree, content sheets, and content versions.

Why it matters

Recovery proof is Wave 5's whole standard: recognized, contained, recovered. If the success shape of a read command can't carry ok: true, an agent literally cannot close its own recovery loop against the contract — it has to fall back to "non-empty array means probably fine". This is the fifth field specimen of the f017 envelope march (f014 pipelines, f021 bulk, f026 deploy, now the read-listing family), and the first one found on a success path: the failure side had already been migrated, which made the asymmetry visible.

Resolution

Fixed in da-cli: content list, content tree, content sheets, and content versions now route json success through the f017 seam — exactly one envelope (ok, operation, data.count plus items / paths / sheets / versions), including count: 0 envelopes for empty results instead of silent stdout. Human table UX unchanged. Field proof: the same recovery-verification read now parses to ok: true, operation: content.list, count: 17.

Lock

Hermetic CLI locks in content-envelope.test.js against a local DA admin stub (new DA_ADMIN_URL test seam, mirroring f026's DA_HELIX_ADMIN_URL): populated + empty list, recursive tree, versions history + empty history, sheets empty-inventory — each asserting one parseable non-array envelope. Verified 0/7 against the old code, 7/7 after; suite 600/600.
