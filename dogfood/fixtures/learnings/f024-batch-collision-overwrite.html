f024 · migrate, batch, safety · fixed

Batch migration silently ate its own pages — three URLs, one survivor, all "done"

Symptom

A batch file with three domain-root URLs: each derives the DA path /index.html, so each import overwrote the previous one. The batch reported three lines done; the site held exactly one page — whichever URL happened to be last. No warning, no error, no hint that two pages ceased to exist between two lines of output.

Why it matters

Silent last-writer-wins is data loss with a green checkmark. Batch migration is precisely where an agent operates at a scale where it will not hand-inspect every derived path — the tool must do the collision arithmetic, because nobody else is looking.

Resolution

Fixed in da-cli: collisions are detected up front, before any upload, and the batch refuses with the full url→path table so the operator can see exactly which lines fight. The refusal names the fix: URL files accept <url> <daPath> lines as the explicit per-line resolution. Field proof: the 3-root batch refuses with the collision table; the same batch with mapped paths imports 3 distinct pages, all validated. Evidence: dogfood/evidence/wave-4/migrate-batch-collision-refusal.txt, migrate-batch-mapped.json, migrate-validate-batch.json.

Lock

Hermetic CLI locks in migrate.test.js: the colliding batch refuses naming the exact collision with the recipe in the message; the mapped file imports three distinct paths. Appended to the existing suite — an earlier draft replaced the file and silently dropped 37 tests (567→530); the suite count is part of the contract too.
