f008 · content · fixed

content merge 404s on a DA folder path

Symptom

After da content clone --path /data, the natural follow-up da content merge /data threw:

DaApiError: DA API 404 at https://admin.da.live/source/{org}/{repo}/data

Merging the workspace with no path (da content merge) succeeded and reported “Merged 2 file(s)”.

Why it matters

Agents chain clone → merge → diff → put-tree. A folder path that works for clone should not 404 on merge, or the CLI should reject the path with a structured recovery. Empty 404 bodies leave the agent guessing.

Root cause

mergeWorkspace treated any path as a single sourceGet target. Clone treats path as a directory root for listing. DA folders are not source documents → empty-body 404. Verb surface looked git/helix-like; path semantics did not match clone (or helix-cli’s file-scoped merge).

Resolution

Fixed in da-cli (src/lib/content-workspace.js): path semantics match clone.

  • No path → refresh entire tracked workspace
  • Document path → that file only
  • Folder / subtree prefix → every document under that prefix (tracked ∪ remote list via listContentPaths)
  • Never sourceGet a bare folder URL
  • Empty / unknown scope → oriented error (not raw 404)

Lock

Unit tests in da-cli/src/lib/content-workspace.test.js (f008 cases: subtree merge, no bare folder sourceGet, document path, new remote under subtree, oriented empty error). Dogfood evidence: re-run da content clone --path /data then da content merge /data must succeed.

0.6.0 friction gate: Critical — closed.
