f026 · deploy, contract · fixed

deploy pages wrote two bare arrays to stdout — a flawless deploy that couldn't be parsed

Symptom

Found deploying the f022–f025 learning pages (the Wave 4 cut's own publish lane — the third session running where publishing the haul generated the next finding): deploy pages --format json printed the preview-phase result array and the publish-phase result array back-to-back. JSON.parse(stdout) failed with Extra data on a perfect 6/6 deploy. No envelope, no ok, no next — just two anonymous lists. Dry-run had the sibling gap: silent-empty stdout.

Why it matters

f014 for pipelines, f021 for bulk jobs, now f026 for deploy: every multi-phase command re-invents output and re-breaks the same contract. The pattern is now unmistakable — the one-command-one-envelope rule must be owned by a seam, not by each author's memory. That is the f017 migration march, and this is its fourth field specimen.

Resolution

Fixed in da-cli: json mode suppresses the per-phase tables and emits one deploy.pages envelope carrying both phases, changes[], errors[] (exit 1 when any path fails); dry-run emits an ok: true, mode: 'dry-run' envelope. Human formats keep both tables. Field proof: the same 6-path deploy re-run parses as one envelope, published: 6/6.

Lock

Hermetic CLI locks in deploy.test.js against a local helix-admin stub (new DA_HELIX_ADMIN_URL test seam): committed single-envelope with both phases, dry-run envelope, and failure→ok:false+exit 1. Verified 0/3 against the old code. Meta-catch: the first lock draft used spawnSync, which starved the in-process stub — the lock's own harness needed the interruption lesson.
