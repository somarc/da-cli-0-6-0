f029 · index, process · recovered

the index config migration was claimed, not observed — deleting the legacy file de-configured live indexing

Symptom

The public query index froze: maxLastModified stuck hours old, freshly published pages never appearing, two wildcard reindex jobs accepted (202) with zero effect. Ground-truth probe: single-path reindex of an already-indexed page matched only #simple — no named index configuration was active anywhere. The previous session had "migrated" index config to the Configuration Service (content/query.yaml), retired the repo-local helix-query.yaml, and updated docs and provenance to say so. But the Configuration Service held no observable config for this site — the identity couldn't even read that surface (403) — so the deleted legacy file had been the active configuration all along. The docs migrated; the platform didn't.

Why it matters

ADR 0002 D6 step 1 is "identify the site's active configuration mode" — observation before assertion. The migration session skipped the observation, and the false claim survived review because it lived in docs, provenance, and a deleted file rather than in failing output. The cost profile is nasty: indexing kept serving (the stored index is a fossil that looks healthy) and only updates died — a silent freshness failure that surfaces days later as "why isn't the new page in the index?". Claims about platform state must carry a live-response proof, exactly like f021's job routes and f016 before it.

Recovery

Recognized: frozen maxLastModified + #simple-only match + config-service 403 triangulated the true state: legacy mode, config deleted. Contained: the stored index kept serving; nothing destructive run against it. Recovered: helix-query.yaml restored from git history and pushed; code sync confirmed codebus parity (changes: [] — the public 404 for the yaml is by design: config files aren't served, a probe-gap lesson in itself); single-path reindex immediately matched the named default index with all custom fields; full index build --wait (through the f028 fix) verified total 39→40, changed: true with the new page present. A real config-service migration remains future work — as an onboarding step with observed proof, not a docs edit.

Lock

Process finding — the lock is discipline, not a unit test: platform-state claims in boards/provenance require a live-response artifact in evidence, and index state has three distinguishable modes (#simple-only = unconfigured; named-index match = configured; frozen public index with accepted jobs = configured-but-stale or de-configured). CLI fitness gap recorded per D6: da index show/validate cannot observe config truth for identities without Configuration Service access — the raw-API fallback used here is the documented workaround.
