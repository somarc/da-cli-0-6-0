f038 · site, config, content safety · fixed

fstab was treated as the source of truth after Configuration Service made it legacy — one false warning hid a system-wide split brain

Symptom

A product-shaped greenfield run began on somarc/throughline, a healthy modern DA-backed EDS site with no repository fstab.yaml. da site info printed three mutually incompatible facts in one result: fstab MISSING, Helix pipeline 200 from markup:https://content.da.live/somarc/throughline/, and non-empty plain HTML — then instructed the agent to commit a mount file. The agent followed the CLI, committed fstab.yaml plus local .da.json, and reverted both minutes later when live platform truth proved the mount already existed.

The narrow warning fix exposed the deeper split brain. site list omitted modern no-fstab sites; site create generated and pushed the legacy file by default; content put and put-tree silently disabled DA structure and section guards when that file was absent; preview diagnostics prescribed it; site doctor bound a public effective-source read to DA auth; and site model fabricated a DA URL instead of observing the configured source.

Why it matters

This was not stale prose around one optional file. The same obsolete predicate controlled both orientation and safety: it could tell an agent to add configuration the platform explicitly says to remove, hide real sites from discovery, and weaken upload validation on the exact modern sites that need it. A single site could therefore be simultaneously “missing,” invisible, healthy at runtime, and exempt from guardrails depending on which command answered.

Configuration Service deliberately separates delivery site, code repository, and content source. Repository file presence cannot be the effective truth. Probe failures, a configured non-DA source, a decoupled DA identity, and a valid legacy declaration are also different states; collapsing them into “no fstab” produces unsafe recovery.

Resolution

First containment: da-cli PR #44 stopped site info from calling an active DA markup source broken merely because fstab was absent.

System fix: PR #45 landed one effective-source contract across the CLI. It reads Helix's effective Sidekick configuration without requiring DA auth (with authenticated fallback for private sites), validates exact content.da.live markup URLs, and distinguishes configured, legacy, unconfigured, and probe-failed. YAML-parsed fstab is consulted only after a conclusive Helix 404; status artifacts can report health but cannot resurrect configuration.

site create is now Configuration Service-first on the valid Adobe boilerplate template; --legacy-fstab is explicit compatibility and site register is the recoverable registration primitive. site list v2 discovers modern sites. site model v2 separates expected from observed source identity. Doctor and preview report effective state without false remediation. Content puts restore strict structure and section checks, including binary target checks; decoupled/prefixed sources block instead of guessing, and unknown source identity requires the explicit --allow-unverified-source ownership flag.

Lock

The old source at cae3e54 was replayed against Throughline: contradictory site info, fstab-only site list, and daBacked:false / strictSections:false are stored under dogfood/evidence/wave-6/f038-before-*. The merged CLI is field-proven in f038-after-*: Throughline appears in site-list.v2; info/model/doctor agree on the same effective source; doctor is seven checks green; and an invalid direct block is refused before any write.

Hermetic coverage spans exact URL/lookalike rejection, scalar and object legacy mounts, modern/legacy/unconfigured/probe-failed precedence, private bearer fallback, decoupled and prefixed write refusal, binary target containment, Configuration Service create/adopt, modern site discovery, parseable JSON contracts, and no-fstab HTML/section enforcement. Final gate: 652/652 tests, lint clean, Node 22 + Node 24 CI green, release smoke green, repository health green; merged as 18877d6.
