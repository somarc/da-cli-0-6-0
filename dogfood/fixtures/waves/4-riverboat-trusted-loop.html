WAVE 4 · TRUSTED LOCAL AUTOMATION · RIVERBOAT GAMBLER

Powerful enough to leave the guardrails

Normal da pipeline YAML is deliberately constrained: DA subcommands only, explicit targets, argv-safe execution, filtered environment keys, and one root --commit boundary. Riverboat Gambler is the supported, intentionally hidden exception for a reviewed local workflow that must also coordinate Git, npm, browser tests, migration scripts, or an outside coding-agent CLI.

Feature-branch field run complete; public promotion pending. The constrained runner refused the fixture, the explicit Riverboat run completed all four steps, the packed CLI smoke passed, and approval/process-tree locks are green. Riverboat YAML is executable code. It runs with the operator's filesystem, network, and ambient credential access. Do not run it from an issue, webpage, model response, untrusted branch, or unreviewed repository. Do not put secrets in YAML or command strings.

What remains after the guardrails are relaxed

The point is not “shell in YAML.” The point is retaining the pipeline engine around trusted local tools:

  • dependency ordering and parallel batches;
  • conditions, interpolation, retries, and reconciliation;
  • explicit requires_approval gates;
  • step timeouts that terminate the spawned process group;
  • persisted run state, soft abort between batches, and one JSON result envelope;
  • DA source, preview, freshness, and code proof as objective verification after local work.

The canonical fixture is dogfood/fixtures/pipelines/riverboat-local-super-loop.yaml. It runs Git and npm as shell steps, then returns to built-in DA freshness and code-verification steps with the explicit pipeline target propagated.

Five workflows this unlocks

  1. Outside-agent repair: call a coding-agent CLI to inspect and patch a page, then run lint, tests, DA preview, and preview explain.
  2. Browser release gate: preview the site, run Playwright, accessibility, screenshots, and visual diffs, then stop at a human approval boundary.
  3. Repository super-loop: combine Git cleanliness, dependency install, lint, unit tests, build, contracts, freshness, and public code proof.
  4. Migration factory: export from a legacy CMS, transform HTML and media with custom scripts, validate, upload to DA, preview, and audit.
  5. Incident runbook: collect DA health, Git history, CI state, and endpoint probes; ask an agent to rank causes; approve one bounded remediation; verify the public result.

These are advanced local compositions, not new responsibilities for da-cli itself. DA remains the verification and operational spine; Riverboat allows adjacent tools to participate in the same observable DAG.
