What Wave 1 is (plain language)

Before any block, sheet, or pipeline claim, the CLI must prove the core loop:

  1. Orientation — status, resolve, site pin-target: which org/repo am I acting on, with which identity, and does that answer survive an omitted flag?
  2. The content loop — content get / put / list / tree and preview page: a fixture in git becomes a page on the preview host, observably.
  3. Diagnosis — site info / doctor / model / freshness: the CLI can describe the site it just touched.
  4. Workspace round-trip — content clone / diff / merge / put-tree: pull content down, prove no drift, push it back without surprises.

The cut rule for every surface: (1) a CLI run, (2) an observable outcome, (3) a file in the evidence pack.

Field notes from the cut

  • Pin first. site pin-target writes .da.json so later commands cannot drift to another project when flags are omitted — the exact failure Wave 5 later injected deliberately (f030).
  • Merge semantics. content merge /data 404s as a DA folder path; merging the cloned workspace root succeeds. Merge workspace contents, not folder URLs.
  • Dry-run is the boundary. put-tree was proven dry-run only — the bulk write path exercised without a single live overwrite.
  • Early findings from this ground: f001 (site info target), f008 (merge path), f009 (put outside worktree).
