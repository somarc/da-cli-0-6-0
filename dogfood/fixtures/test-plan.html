WAVES · CHECKPOINTS · GATE

The test plan

Every command family da-cli ships is classified and exercised against this site. Nothing escapes coverage: adding a command without a place in this plan is itself a failure. The plan runs in waves, each building on the proven foundation of the last.

What "done" means for a check

A check passes only when it asserts an observable state transition and proves it publicly — the DA source changed, the preview activated, the live URL renders the expected content, the route classifies as expected, the job reached a terminal state. Retrying a command until it exits zero is not evidence. Counting commands is not a score.

The waves

  • Wave 1 — Foundation & core loop. Config, status, resolve, content CRUD, preview, publish, site health, freshness, local serving. The end-to-end author-then-prove loop. (In progress — this very page is part of it.)
  • Wave 2 — Blocks, audits, design. Pages built from real blocks, with contract and semantics audits proving the block code resolves on the served branch.
  • Wave 3 — Structured data & routes. A DA sheet, nested pages, and deliberate route cases: content-owned, code-owned, hybrid, orphan, and probe-failed. Index configured through the Configuration Service.
  • Wave 4 — Coordination, durability, migration. Pipelines, durable jobs that survive interruption, and a migration with a known expected result.
  • Wave 5 — Failure & recovery. Deliberately induced failures — source drift, missing assets, stale delivery, protected routes, cancelled jobs — each recognized, contained, and recovered.
  • Wave 6 — Lifecycle & conditional. Auth, skills, a second site created from scratch, and credentialed integrations where available.

The gate

When every required surface is green with stored evidence, the construction pipeline regenerates this site, and provenance verifies — then, and only then, da-cli 0.6.0 is cut.
