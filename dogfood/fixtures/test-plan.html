The test plan

Every command family da-cli ships is classified and exercised against this site. Nothing escapes coverage: adding a command without a place in this plan is itself a failure. The plan runs in waves, each building on the proven foundation of the last.

Last updated: 2026-07-17 01:50 UTC · keep this stamp current whenever the plan or progress log changes.

What "done" means for a check

A check passes only when it asserts an observable state transition and proves it publicly — the DA source changed, the preview activated, the live URL renders the expected content, the route classifies as expected, the job reached a terminal state. Retrying a command until it exits zero is not evidence. Counting commands is not a score.

Progress on this page must stay honest: when work lands, append the progress log with a UTC timestamp. Do not mark a wave cut without evidence under dogfood/evidence/ (or an equivalent public proof).

The waves

  • Wave 1 — Foundation & core loop. Config, status, resolve, content CRUD, preview, publish, site health, freshness, local serving. Author-then-prove loop. Board: git dogfood/WAVES.md · evidence: dogfood/evidence/wave-1/.
  • Wave 2 — Blocks, audits, design. Field-scale block spectrum on /kitchen-sink (see dogfood/BLOCK-COVERAGE.md). Contracts + full audit + design surfaces. Worker-backed blocks use local wrangler stubs (workers/README.md), not production edge secrets.
  • Wave 3 — Structured data & routes. DA sheets, nested pages, route ownership cases (content, code, hybrid, orphan). Index via Configuration Service / da index build.
  • Wave 4 — Coordination, durability, migration. Pipelines, durable jobs that survive interruption, migration with known expected result.
  • Wave 5 — Failure & recovery. Induced failures — source drift, missing assets, stale delivery, protected routes, cancelled jobs — each recognized, contained, and recovered.
  • Wave 6 — Lifecycle & conditional. Auth, skills, a second site from scratch, credentialed integrations where available.
