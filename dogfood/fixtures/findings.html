FLYWHEEL · FIELD · FIX

Findings

Dogfooding earns its keep by surfacing real problems that unit tests cannot. The discipline here is a flywheel: a problem found while operating a real site becomes a primitive fix with a regression test, so the tool gets safer every time it is used in the field. The more the CLI is dogfooded, the smarter and more trustworthy it becomes.

Finding #1 — site info ignored the target override

On the very first orientation command against this site, da site info reported pipeline health for the wrong repository. The command resolved its target from a stray config file left in the working directory and silently ignored the explicit --org and --repo flags — the one thing an operator relies on to say "no, I mean this site."

Every other command honored the override; this one had drifted onto its own resolution path. Left unnoticed, it is exactly the class of bug that teaches an agent not to trust the interface.

Resolution: route site info through the shared config resolver so the override is sovereign everywhere, and add a regression test that locks it: root flags are honored, and explicit arguments win. The fix shipped with the release this site certifies.

Finding #2 — design detect gave no answer on a design-less site

Running da design detect against a site with no design system exited with an error and printed nothing to standard output. But there is a real difference between "the command failed" and "there is no design system here," and an agent cannot tell them apart from an empty error.

Resolution (planned): return a structured result for the no-design-system case — a clean envelope that says so — rather than an empty error. This is the principle the whole release is built on: every operation should leave the agent better oriented, including when the answer is "nothing found."

Finding #3 — the query index cannot be built through da

Wave 3 makes the coverage page index-driven: each page declares its wave, surface, and status as metadata, and the query index aggregates them. Building that index surfaced two coupled gaps. First, the mechanism: Edge Delivery populates the query index from published content only — previewing a page never adds it to the index, and re-authoring a published page does not update its index record until the page is published again. The index is a projection of the live site, not the preview.

Second, the tool: the da index family reads the index (show, validate, query) but cannot cause it to change. There is no primitive to reindex, and nothing tells an agent that its freshly-previewed metadata will not appear until publish. So an agent that authors index-driven content with da-cli hits a wall — the tool can describe the index but cannot make it reflect reality — and falls back to raw curl against admin.hlx.page/index/…, exactly the fallback the tool exists to eliminate.

Resolution (planned): teach da publish to report the index consequence of a publish, add a da index build reindex primitive over the admin endpoint, and have da index show/validate/query state plainly that the index reflects published state. Then the construction pipeline can publish-and-index its own coverage data, and the index-driven ToC becomes provably regenerable like every other artifact.

Finding #4 — index query --env preview silently queries live

While exercising the index surface, da index query default --env preview returned live results, not preview. The command had kept a local --env alias for --target, but root --env already means DA Admin API environment (prod|stage|dev). The value was swallowed before the index command could see it, defaulting to live with a plausible URL — same trust class as finding #1.

Resolution: drop the index-local --env alias entirely. Index validate/query resolve the host only via --target preview|live. If root --env is mistakenly set to preview or live, the CLI hard-errors with recovery text pointing at --target. Locked by unit tests on resolveIndexQueryHost.

Finding #5 — route classify calls live code assets "orphan"

Classifying a real, served code asset — da route classify /scripts/aem.js, which returns 200 on the code bus — reported orphan. The classifier infers ownership only from a content route's sourceLocation (is it content.da.live or a code repo?). It never consults the code-bus status the admin API already returns in the same response — where this asset shows code.status: 200. With no content-route source location, a live code file falls through every branch to "orphan."

"Orphan" is the verdict that invites cleanup, so a route tool that labels live code assets orphaned is quietly steering toward the wrong action. The fix is small and the signal is already in hand: read code.status from the status response and classify a path with code presence and no DA source as codebus. Until then, codebus is only ever returned for hybrid pages rendered from code templates, not for the code bus itself.

Resolution (planned): fold the admin code status into the classifier so codebus and hybrid reflect actual code-bus presence, with a locking test over a known code asset.

Finding #6 — a deleted source still classifies as "contentbus"

Wave 3 built a deliberate orphan: publish a page, then delete its DA source, leaving a live artifact with no owner. That is the exact case route classify exists to flag. It missed it — the verdict was contentbus, even though the same output carried daSource: false. The classifier trusts the sourceLocation that the preview/live status still reports (it points at content.da.live from the last activation) over its own probe that just proved the source is gone.

An agent told "contentbus" believes the source exists and is safe to edit, when in fact there is nothing to edit and a stale live artifact to remove. The fix is the same shape as finding #5: when daSource is false but a live artifact persists, that is an orphan, not content — the corroborating signal is already in the response and should win over the stale location.

Finding #7 — route clean does not remove the orphan

Running da route clean on that orphan deleted the (already-gone) DA source and attempted a best-effort preview flush that is a documented no-op in production. It never unpublished the route. Live and preview both stayed 200: the artifact the command exists to remove survived cleaning. A separate publish unpublish was also required — and returned 403 with the operator's current token, so the orphan could not be removed at all.

Two gaps compound here. First, route clean should unpublish the live route and delete the preview artifact through the admin API, not rely on a flush that does nothing in prod — otherwise "clean" leaves the mess in place. Second, an unpublish that is refused by permissions should surface as a structured, recoverable result (this token cannot unpublish; here is who can), not a bare 403. This orphan remains live as its own evidence until it can be properly removed.

How findings are recorded

Each finding is logged with a stable reference, the surface it appeared on, the resolution, and the test that prevents its return. The living list travels with the construction sources so it is never separated from the site it describes.
